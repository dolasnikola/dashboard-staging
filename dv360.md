# Plan: Migracija DV360 na Supabase Edge Function

## 1. Pregled trenutnog stanja

**Sadašnji flow:**
```
DV360 → Email CSV report → Gmail → Apps Script čita email → Supabase REST API
```

**Problemi:**
- Zavisi od email delivery-ja (krhko, kašnjenja, spam filter)
- Apps Script nema dobar error handling / retry
- Nema centralizovanog logovanja (sync_log)
- Hardkodirani filteri u kodu (exclude Farma/Pharm/Septolete)
- Nema backfill mogućnosti (date range override)

**Ciljni flow:**
```
pg_cron → Edge Function "sync-dv360" → DV360 Reporting API → Supabase
```

---

## 2. Preduslov: DV360 API pristup

| Korak | Detalj |
|-------|--------|
| **Google Cloud Project** | Koristiti postojeći ili napraviti novi u Google Cloud Console |
| **Uključiti API** | Enable "Display & Video 360 API" (`displayvideo.googleapis.com`) i "DV360 Reporting API" (`doubleclickbidmanager.googleapis.com`) |
| **Service Account** | Kreirati Service Account u Google Cloud, preuzeti JSON key |
| **DV360 pristup** | U DV360 UI (Advertisers > Users) dodati Service Account email kao korisnika sa read pristupom |
| **Supabase Secret** | `supabase secrets set DV360_SERVICE_ACCOUNT_JSON='{"type":"service_account",...}'` |

> **Napomena:** Za razliku od Google Ads, DV360 API **ne zahteva** developer token review proces. Service Account sa pristupom je dovoljan.

---

## 3. Arhitektura Edge Function-a

Struktura fajlova (isti pattern kao sync-meta i sync-gemius):

```
supabase/functions/sync-dv360/
├── index.ts          — Entry point (CORS, trigger parsing, timezone, sync_log, config loop)
├── auth.ts           — Google OAuth2: JWT signing → access_token (jose library)
├── api.ts            — DV360 Reporting API pozivi
├── sync-client.ts    — Logika po klijentu: fetch → transform → upsert
└── types.ts          — TypeScript tipovi
```

## 4. Detaljan opis fajlova

### 4.1 `types.ts`

```typescript
interface DV360Config {
  id: number;
  client_id: string;          // npr. "krka"
  advertiser_id: string;      // DV360 advertiser ID
  partner_id: string;         // DV360 partner ID
  enabled: boolean;
  last_synced_at: string | null;
}

interface CampaignRow {
  client_id: string;
  platform: "dv360";
  date: string;
  campaign: string;
  insertion_order: string;
  impressions: number;
  clicks: number;
  spend: number;
  reach: number;
  conversions: number;
  conv_value: number;
  ctr: number;
  cpm: number;
  cpc: number;
  cpa: number;
}

interface SyncResult {
  clientId: string;
  platform: "dv360";
  status: "ok" | "error";
  rows?: number;
  months?: string[];
  error?: string;
}
```

### 4.2 `auth.ts` — Google OAuth2 za Deno

```
1. Parsirati Service Account JSON iz env varijable
2. Kreirati JWT sa:
   - iss: service account email
   - scope: "https://www.googleapis.com/auth/doubleclickbidmanager https://www.googleapis.com/auth/display-video"
   - aud: "https://oauth2.googleapis.com/token"
   - exp: +1 sat
3. Potpisati JWT sa RS256 (privatni ključ iz JSON-a) koristeći `jose` library
4. POST na https://oauth2.googleapis.com/token sa grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer
5. Vratiti access_token (važi 1 sat, dovoljno za ceo sync)
```

> **Ključna biblioteka:** `import * as jose from "npm:jose@5"` — radi u Deno, podržava RS256.

### 4.3 `api.ts` — DV360 Reporting API

**Pristup — DBM Reporting API:**

```
1. POST https://doubleclickbidmanager.googleapis.com/v2/queries
   - Kreirati query sa:
     - type: STANDARD
     - metadata.dataRange: CUSTOM_DATES (date_from, date_to)
     - params.type: STANDARD
     - params.groupBys: [FILTER_DATE, FILTER_ADVERTISER, FILTER_INSERTION_ORDER, FILTER_LINE_ITEM]
     - params.metrics: [METRIC_IMPRESSIONS, METRIC_CLICKS, METRIC_REVENUE_ADVERTISER, METRIC_TOTAL_CONVERSIONS, METRIC_RICH_MEDIA_VIDEO_COMPLETIONS]
     - params.filters: [{ type: FILTER_ADVERTISER, value: advertiser_id }]

2. POST https://doubleclickbidmanager.googleapis.com/v2/queries/{queryId}:run
   - Pokrenuti report

3. GET https://doubleclickbidmanager.googleapis.com/v2/queries/{queryId}/reports
   - Polling dok report ne bude gotov (status: DONE)
   - Obično 10-30 sekundi

4. GET report download URL (Google Cloud Storage link)
   - Preuzeti CSV rezultat

5. DELETE query nakon preuzimanja (cleanup)
```

### 4.4 `sync-client.ts` — Logika po klijentu

```
async function syncDV360Client(supabase, config, accessToken, dateOverride?):
  1. Odrediti date range:
     - Default: juče (kao sync-meta)
     - Manual: dateFrom/dateTo iz request body

  2. Kreirati DV360 report query (api.ts)

  3. Pokrenuti report i čekati rezultat (polling, max 2 min timeout)

  4. Parsirati CSV rezultat

  5. Svi podaci se unose u bazu — BEZ filtriranja na sync nivou.
     Filtriranje (npr. "krka terme" vs "Farma/Pharm") se radi u frontendu.

  6. Transformacija u CampaignRow format:
     - campaign = Line Item name ili Campaign name
     - insertion_order = Insertion Order name
     - spend = Revenue (Advertiser Currency)
     - Izračunati: ctr, cpm, cpc, cpa

  7. Deduplikacija (isti pattern kao sync-meta)

  8. Upsert via RPC upsert_campaign_data_by_dates
     - Isti RPC kao sync-meta (campaign_data tabela, isti format)

  9. Update last_synced_at u dv360_config

  10. Return SyncResult
```

### 4.5 `index.ts` — Entry point

Identičan pattern kao sync-meta/sync-gemius:
- CORS handling
- Trigger parsing (cron / manual sa date override)
- Belgrade timezone check (8:00, 9:00)
- sync_log kreiranje
- Fetch `dv360_config` tabele
- Google OAuth2 token (auth.ts) — jednom za sve klijente
- Sekvencijalno po klijentu
- sync_log update
- JSON response

## 5. Filtriranje podataka

**Odluka:** Svi DV360 podaci se unose u `campaign_data` tabelu bez filtriranja.

Filtriranje (npr. prikazivanje samo "krka terme" kampanja, isključivanje Farma/Pharm/Septolete) se radi isključivo na frontend nivou. Ovo omogućava:
- Potpune podatke u bazi za sve potrebe (reporting, analiza)
- Fleksibilnu promenu filtera bez ponovnog sync-a
- Jednostavniju Edge Function logiku (nema hardkodiranih filtera)

Frontend već koristi `insertion_order` polje za DV360 podatke — filtriranje se implementira u `getFilteredData()` ili na komponenti nivou.

## 6. Nova tabela: `dv360_config`

```sql
CREATE TABLE dv360_config (
  id SERIAL PRIMARY KEY,
  client_id TEXT NOT NULL REFERENCES clients(id),
  advertiser_id TEXT NOT NULL,
  partner_id TEXT NOT NULL,
  enabled BOOLEAN DEFAULT true,
  last_synced_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- RLS
ALTER TABLE dv360_config ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Service role full access" ON dv360_config
  FOR ALL USING (auth.role() = 'service_role');
```

## 9. Rizici i mitigacija

| Rizik | Mitigacija |
|-------|------------|
| DV360 Reporting API polling timeout | Max 2min čekanje, retry jednom, log error u sync_log |
| Service Account nema pristup | Testirati ručno pre deploy-a, jasna error poruka |
| Report API rate limits | Sekvencijalno po klijentu, jedan report istovremeno |
| Insertion order polje prazan | Fallback na campaign name (isti pattern kao sad u `data.js`) |
| Razlike u podacima vs Apps Script | Paralelno pokretanje oba sistema 3-5 dana, uporediti podatke |

---

## 10. Prednosti nakon migracije

- **Eliminisan email dependency** — direktan API, nema Gmail parsiranja
- **Svi podaci u bazi** — nema filtriranja na sync nivou, fleksibilnost za analizu
- **Backfill** — `POST {"trigger":"manual","date_from":"...","date_to":"..."}` za bilo koji period
- **Centralizovani logovi** — sve u `sync_log` tabeli
- **Konzistentnost** — isti pattern kao sync-meta i sync-gemius
- **Alerting** — `check-alerts` Edge Function već detektuje sync failure
